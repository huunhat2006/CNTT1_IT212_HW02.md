Ý nghĩa của Stack Trace & Mã lỗi H2
org.h2.jdbc.JdbcSQLIntegrityConstraintViolationException: Đây là ngoại lệ do database H2 ném ra khi một ràng buộc toàn vẹn (Integrity Constraint) của cơ sở dữ liệu bị vi phạm trong quá trình thực thi câu lệnh SQL.

Mã lỗi [23506-214]: Mã lỗi SQL State 23506 là mã chuẩn hóa cho lỗi Referential Integrity Violation (Vi phạm toàn vẹn tham chiếu). Số 214 phía sau là phiên bản của H2 Database đang sử dụng.

Thông báo chi tiết: "CONSTRAINT_FOREIGN_KEY_USER: PUBLIC.TRANSACTIONS FOREIGN KEY(USER_ID) REFERENCES PUBLIC.USERS(ID) (99)" chỉ ra chính xác rằng bạn đang cố gắng INSERT một bản ghi vào bảng TRANSACTIONS với USER_ID = 99, nhưng giá trị 99 này lại không tồn tại trong cột ID của bảng USERS.

Foreign Key Constraint là gì và Vì sao lỗi xảy ra?
Khóa ngoại (Foreign Key) là một ràng buộc được thiết lập giữa hai bảng nhằm đảm bảo tính đồng nhất và toàn vẹn của dữ liệu. Nó không cho phép bảng con (TRANSACTIONS) chứa các giá trị tham chiếu đến bảng cha (USERS) mà không tồn tại ở bảng cha.

Nguyên nhân gốc rễ (Root Cause): Do tầng ứng dụng (Application Layer) đã bỏ qua bước validate dữ liệu đầu vào (hoặc thiếu kiểm tra business logic). Hệ thống tin tưởng tuyệt đối vào tham số userId truyền lên từ Client và đẩy thẳng xuống DB. Khi DB kiểm tra và phát hiện userId mồ côi (orphan), nó buộc phải từ chối hành động này bằng cách ném ra lỗi để bảo vệ dữ liệu.

2. Đề Xuất Hướng Sửa Đổi
Thay vì chỉ đặt một khối try-catch bọc xung quanh câu lệnh INSERT để bắt SQLException (điều này rất thụ động và có thể che giấu các lỗi DB khác như mất kết nối, lỗi syntax), chúng ta sẽ chủ động phòng ngừa (Defensive Programming) bằng cách:

Bước 1 (Check): Thực hiện một câu lệnh SELECT COUNT(1) hoặc SELECT id từ bảng USERS với userId được cung cấp.

Bước 2 (Validate): Nếu kết quả trả về bằng 0 (không tồn tại), ứng dụng chủ động ném ngay một Custom Business Exception (UserNotFoundException).

Bước 3 (Action): Nếu người dùng hợp lệ, hệ thống mới tiếp tục thực thi câu lệnh INSERT.

Viết lại Class TransactionRepository
Java
package com.securebank.repository;

import com.securebank.exception.UserNotFoundException;
import java.sql.Connection;
import java.sql.PreparedStatement;
import java.sql.ResultSet;
import java.sql.SQLException;

public class TransactionRepository {

    // Connection nên được quản lý bởi Connection Pool (như HikariCP) và inject vào qua Constructor/Method
    private final Connection connection;

    public TransactionRepository(Connection connection) {
        this.connection = connection;
    }

    /**
     * Lưu thông tin giao dịch an toàn sau khi đã kiểm tra sự tồn tại của User.
     *
     * @param transactionId Mã giao dịch
     * @param userId        ID người dùng
     * @param amount        Số tiền giao dịch
     * @throws UserNotFoundException Nếu không tìm thấy user tương ứng trong DB
     * @throws SQLException          Nếu có lỗi kết nối hoặc truy vấn DB xảy ra
     */
    public void saveTransaction(String transactionId, Long userId, double amount) 
            throws UserNotFoundException, SQLException {
        
        // 1. Kiểm tra sự tồn tại của User trước khi thực hiện nghiệp vụ
        if (!isUserExists(userId)) {
            throw new UserNotFoundException("Không thể thực hiện giao dịch. Người dùng có ID " + userId + " không tồn tại.");
        }

        // 2. Chuẩn bị câu lệnh INSERT dữ liệu
        String insertSql = "INSERT INTO transactions (id, user_id, amount) VALUES (?, ?, ?)";

        // Sử dụng try-with-resources (Java 7+) để tự động giải phóng PreparedStatement
        try (PreparedStatement psInsert = connection.prepareStatement(insertSql)) {
            psInsert.setString(1, transactionId);
            psInsert.setLong(2, userId);
            psInsert.setDouble(3, amount);

            psInsert.executeUpdate();
        }
    }

    /**
     * Phương thức helper kiểm tra xem User có tồn tại trong DB hay không.
     */
    private boolean isUserExists(Long userId) throws SQLException {
        if (userId == null) {
            return false;
        }

        String checkUserSql = "SELECT 1 FROM users WHERE id = ?";
        
        // Sử dụng try-with-resources để tự động đóng PreparedStatement và ResultSet
        try (PreparedStatement psCheck = connection.prepareStatement(checkUserSql)) {
            psCheck.setLong(1, userId);
            
            try (ResultSet rs = psCheck.executeQuery()) {
                // Nếu ResultSet có dữ liệu (next() == true), tức là user có tồn tại
                return rs.next();
            }
        }
    }
}
5. Vì Sao Phiên Bản Mới An Toàn Hơn?
Phiên bản mới mang lại nhiều giá trị vượt trội cho hệ thống production nhờ các lý do sau:

Chuyển đổi từ lỗi kỹ thuật sang lỗi nghiệp vụ (Business Error): Việc ném ra UserNotFoundException giúp tầng Controller/API dễ dàng bắt được và trả về mã lỗi HTTP đẹp mắt cho Client (ví dụ: 404 Not Found đi kèm thông điệp rõ ràng), thay vì phơi bày lỗi 500 Internal Server Error kèm theo Stack trace thô của Database ra ngoài, gây mất an toàn an ninh thông tin.

Tránh rò rỉ tài nguyên hệ thống (Resource Leak): Phiên bản cũ không hề đóng PreparedStatement. Nếu hệ thống có lượng giao dịch lớn (High Throughput), việc này sẽ nhanh chóng làm cạn kiệt bộ nhớ và pool của database, dẫn tới sập hệ thống (OOM hoặc Connection Timeout). Việc áp dụng try-with-resources đảm bảo mọi tài nguyên mở ra đều được đóng lại một cách tuyệt đối, kể cả khi có Exception xảy ra.

Đảm bảo tính nhất quán dữ liệu & Giảm thiểu DataIntegrityViolationException: Trong các hệ thống lớn sử dụng JPA/Hibernate bọc ngoài JDBC, một khi DB ném ra DataIntegrityViolationException, Transaction hiện tại sẽ lập tức bị đánh dấu là Rollback-only. Điều này có nghĩa là bạn không thể cứu vãn hay thực hiện thêm bất kỳ hành động ghi nhận (như ghi Log lỗi vào DB, cập nhật trạng thái transaction thất bại...) trong cùng transaction đó nữa. Kiểm tra chủ động ở tầng ứng dụng giúp giữ cho Transaction "sạch", cho phép lập trình viên toàn quyền điều hướng luồng xử lý lỗi mà không làm ảnh hưởng đến trạng thái của kết nối DB.
